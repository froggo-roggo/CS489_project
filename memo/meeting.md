### 진행상황

#### 워터마크 기반

- 서버가 없을 때도 HTML 문서 내에서만 인코딩/디코딩하는 유사한 작업을 발견했다.
  
  - 문서나 코드 중요한 부분 숨기거나 암호화
  
  - 개발자 도구에서 HTML 문서 내용 숨기기 등등

- 기존 방식
  
  - 복사 버튼을 누르면 event로 클립보드에 다른 글(워터마크) 삽입
  
  - 투명도 0%인 텍스트를 글 사이에 넣어 복사나 크롤링을 방지
    
    - 다만 이 경우 스크린리더에서 투명한 텍스트가 그대로 읽힐 수 있음
  
  - JavaScript 기반의 방식은 특정 부분을 제외해서 작동을 막는 식으로 회피되곤 함 

- 목표
  
  - JavaScript를 가능한 안 쓰고 본문에서만 크롤링을 방지하기

- 기존 방식과의 차별점
  
  - PHP나 JavaScript로 인코딩해서 하는 게 아니라, Python parser와 비슷하게 문서를 넣어주면 알아서 적절한 위치에 워터마크를 끼우는 식으로, HTML에 문외한이라도 브라우저 확장 프로그램 식으로 만들면 사용하기 편할 듯.

#### 암호화 기반

- 공개키-개인키 기반 방식으로 개인이 원하는 키로 암호화할 순 있으나, 복호화 과정이 안 드러나게 웹에서 기능 구현하는 것이 가능한지 더 조사해야 한다.

- 기존 방식
  
  - HTML 태그를 읽고 원하는 부분을 크롤링하는 걸 막기 위해 태그 이름 자체를 암호화하는 방식 제안된 바 있음
  
  - Domain 자체를 키로 사용해서, 내가 원하지 않는 곳에서 읽히는 것을 막는 방식

### 정리

1. 워터마크 자동화 프로그램
   
   1. Python 기반으로 텍스트 파트를 찾아서 사이에 워터마크 문구 삽입
   
   2. 이때 레이아웃이 안 망가지도록 고려해야 함

2. HTML 텍스트 암호화
   
   1. Javascipt 기반으로 텍스트 셔플링
   
   2. 도메인 제한을 걸 수도 있음

3. 접근성 유의
   
   1. 기존 방식들은 문제 제시만 하고 해결책은 아직 없음
   
   2. 전용 스크린 리더나, 웹 접근성 진단 같은 방향으로 나가도 괜찮을 듯
   
   3. 기존 서비스를 개선하는 쪽으로 나가도 좋을 듯

### 아이디어

- 이미지 배너 광고의 접근성 향상 by 정원빈
  
  - 스크린리더 얘기가 나왔어서 그런데, 광고를 읽어주는 스크린 리더도 가능하지 않을까?

- 시각 장애인용 광고차단 by 정원빈
  
  - 시각 장애인은 대다수가 전맹이 아닌 저시력자라, 화면 확대기나 스크린 리더를 같이 쓰는데 팝업 광고의 경우 대응하기 어렵다.
  
  - 이를 개선하는 프로그램이 있을까? 아예 anti-AD를 쓰는 방법도 있겠지만, 늘 가능한 방식은 아니다.

- 최악의 광고 모음 by 양지수
  
  - 최악의 UI, 패스워드 설정하는 게임 같이 솔루션이 아니라 재미있는 경험을 제공하는 프로젝트도 괜찮을 거 같다.
  
  - 우리의 경우 최악의 접근성을 제공하거나 최고의 악질 팝업 광고 제작 등이 가능할 듯하다.

### 주제 재선정

팝업의 경우 아예 새 창을 띄우거나 부분/전체화면을 차지하고 본문과의 상호작용(스크롤, 클릭)을 막는 경우가 있다. Anti-AD를 써도 오히려 그런 류의 프로그램을 해제해달라고 요청을 띄우는 경우가 있고, 광고를 못 막기도 한다. 주로 `style` 태그 내의 `z_indx`를 써서 레이어 순위 조정하는 식이 주된 방식이다. 이 경우는 `z_indx`의 값만 마이너스로 바꾸면 되는데, Google Adsense가 제공하는 전형적인 케이스가 아니어도 수정할 수 있다.

접근성 개선을 위해,

1. 팝업창이 뜨는 경우 알림을 띄우기

2. 팝업창 끄거나 확인하는 버튼 강조하기

3. 자체적인 팝업창 제거 선택지 제공하기

등등의 기능을 구현하면 좋을 거 같다. 이때 팝업을 아예 막아버리면 생기는 문제가 어떤 경우가 있는지 edge case 고민해야 한다.

        - 대안: 대신 버튼 요소 위에 따로 div나 layer를 올려서 opacity 사용해서 강조할 순 없나?
    
    - 대안: 이 페이지 요소는 팝업입니다/여기서 팝업이 끝납니다를 팝업 요소에 body로 넣어서 스크린리더가 읽게 만들기

Problem Flow

1번 페이지 이후 모든 문제 페이지에서 주기적으로 팝업 띄우기

등장하는 모든 이미지들에 대해 로딩 지연

1. Introduction
   
   1. 프로젝트에 대한 간단한 설명
   
   2. 위의 메뉴 바에 대한 설명
   
   3. 지시: 가능한 색맹/색약/저시력자 옵션 중 하나 고르고, 마우스 대신 tab 키와 방향키 사용할 것을 권장
   
   4. start 버튼 눌러서 시작

2. 듣기 문제: 팝업과 뒤섞인 HTML 문서 문제
   
   1. 주제: HTML 레이아웃
   
   2. 음성 내용: 멀쩡한 문서 모양와 대조되는 엉망인 스크린 리딩 음성  
      내용은 HTML 레이아웃에 대한 것
   
   3. 문제: 30초 부근에서 읽고 있는 내용은 몇 번째 문단의 내용인가?
   
   4. 힌트: 처음엔 문서 내용을 보여주지 않고 힌트 눌러야 보여줌
   
   5. 방해 요소: 순서가 뒤섞인 HTML 레이아웃, 스크린 리더에서 읽히는 팝업 광고 내용

3. 읽기 문제: 텍스트 읽고 문제 풀기
   
   1. 주제: 텍스트 대비
   2. 텍스트 내용: 웹 접근성의 개념과 예시(텍스트 대비)에 대한 글
   3. 문제: 웹 접근성의 개념에 부합하는 설명은?  
      보기: 1, 2, 3, 4를 각각 다른 색의 글씨로 작성
   4. 방해 요소: 너무 작은 글씨, 읽기 어려운 글꼴, 배경색과 글자색 구분 어렵게 지정, 페이지에서 너무 많은 색 사용

4. 특정 업무 수행: 원하는 내용만 찾아서 읽기
   
   1. 주제: 제목 구조
   2. 내용: 계층 구조를 가진 장문의 글
   3. 문제: 특정한 주제 관련된 장을 찾아서 거기의 7번째 문장 복사 붙여넣기
   4. 힌트: 제목 구조 제공
   5. 방해 요소: Heading으로 제공되지 않는 제목 구조

5. 특정 업무 수행: 키보드만 사용해서 버튼 누르기
   
   1. 주제: 키보드 접근성
   2. 내용: 각종 키보드 접근성용 key에 대한 글
   3. 문제: tab 키만 사용해서 특정 버튼 3개 순서대로 누를 경우 완료
   4. 방해 요소: 요소 강조 표시 눈에 잘 안 띄게 만들기, 요소의 순서 뒤바꿔두기

6. 듣기 문제: 스크린리더 상의 문제 체험
   
   1. 주제: 부정확하거나 없는 대체 텍스트
   2. 음성 내용: 스크린 리더와 대체 텍스트에 관한 이야기를 읽어줌  
      보안 규칙에 대한 이야기 및 인증용 이미지 예시
   3. 문제: 들은 보안 규칙에 부합하는 패스워드를 만들고 이미지의 인증 코드를 입력하시오.
   4. 힌트: 힌트 버튼 누르면 음성 대신 실제 화면 확인 가능
   5. 방해 요소: 이미지를 잘못 설명하는 대체 텍스트, 지나치게 길고 장황한 대체 텍스트, 장식용 이미지에도 사용된 대체 텍스트, 캡션과 대체 텍스트 동시 사용(동일한 설명 두 번 듣게 됨), 분명하지 않은 링크(이미지를 링크로 사용, 밑줄친 푸른 글자에 링크 안 걸기)

7. 듣기 문제: 이미지 링크 찾아서 누르기
   
   1. 주제: 링크 텍스트 누락 & 모호한 링크 텍스트
   2. 음성 내용: 여러 개의 이미지와 링크 텍스트 있는 페이지의 스크린 리딩 내용
   3. 문제: 다음 단계로 가는 올바른 링크를 찾아 눌러 이동하세요.
   4. 힌트: 힌트 버튼 누르면 음성 대신 실제 화면 확인 가능
   5. 방해 요소: 이미지에 대체 텍스트로 로고라고만 설명하고 링크란 설명 안 넣기, 어디로 이동하는지 모르는 여기를 클릭하세요/자세히 보기 등의 링크, 파랗게 밑줄만 쳐 있고 링크는 아닌 텍스트

맨 마지막 완료 페이지에서 감사인사+최악의 패턴 투표
